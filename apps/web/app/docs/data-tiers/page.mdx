# Data Tiers

The engine is given its ephemeris data explicitly. Two suppliers ship with the
package: the **embedded** dataset for the browser and edge, and the **Node
loader** for the extended packs. The hosted [REST demo](/api/chart?lat=27.94&lon=-82.46)
and [MCP server](/docs/mcp) build on these two.

## What runs where

| Capability | Embedded (browser, edge) | Node loader | REST `/api/chart` | MCP (`caelus-mcp`) |
| --- | :---: | :---: | :---: | :---: |
| Core bodies (Sun–Pluto, Chiron, nodes) | ✓ | ✓ | ✓ | ✓ |
| Houses (12 systems), angles, aspects | ✓ | ✓ | Placidus | ✓ |
| Tropical + 7 sidereal ayanamsas | ✓ | ✓ | tropical | ✓ |
| Derived charts (returns, progressions, …) | ✓ | ✓ | — | transits, synastry |
| Extended bodies (asteroids, Uranian points) | — | ✓ | — | ✓ |
| Fixed stars (318-star catalog) | ✓ | ✓ | ✓ | ✓ |
| Event search (rise/set, crossings, phases, stations) | — | ✓ | — | ✓ |
| Eclipses (solar, lunar) | — | ✓ | — | ✓ |
| Data on disk | none (bundled) | JSON packs (precise Moon ~3.1 MB + catalogs) | none (edge) | JSON packs (Node) |
| Date range | 1800–2149¹ | 1800–2149¹ | 1800–2149 | 1800–2149¹ |
| Accuracy | per body, see [Validation](/validation) | per body, see [Validation](/validation) | per body | per body |

<Note>
  ¹ 1850–2150 for the precise Moon and Chiron tiers. The embedded dataset
  computes everything from the bodies it bundles; the Node tier adds the data
  the extended features were validated against. Anything pure-arithmetic on
  positions, such as derived charts and sidereal shifts, runs in any tier,
  including the browser. See <a href="/docs/edge-cases">Edge Cases &amp; Stability</a> for the
  out-of-range and missing-pack behaviour.
</Note>

## Embedded (browser, edge)

`caelus/data-embedded` bundles a compact dataset: the eight planets, the Moon,
Pluto, Chiron, the nodes, and the HYG fixed-star catalog (~13 KB gzipped of the
~98 KB total), with the core house systems and aspects. This is what runs
client-side and on edge runtimes, with no files on disk.

<CodeBlock
  lang="typescript"
  label="embedded.ts"
  code={`import { Engine } from "caelus";
import { embeddedData } from "caelus/data-embedded";

const engine = new Engine(embeddedData);`}
/>

## Node loader (extended)

`loadNodeData` reads the larger JSON packs from disk: higher-precision VSOP
tiers, the precise Moon pack, a wide-range Pluto pack (a Chebyshev fit to the
JPL Horizons barycenter that supersedes the embedded Meeus series and holds
across 1850&ndash;2150), the asteroids and Uranian points. Fixed stars ship in
embedded too; the Node loader reads the same catalog from disk when you load
extended packs. Use the Node tier when you need extended bodies or event search.

<CodeBlock
  lang="typescript"
  label="node.ts"
  code={`import { Engine } from "caelus";
import { loadNodeData } from "caelus/node";

const engine = new Engine(loadNodeData());
const star = engine.fixedStar("Aldebaran", jd);`}
/>

<Note>
  If a pack is missing, <code>loadNodeData</code> falls back per body to the
  embedded tier rather than throwing. Packs are versioned JSON build outputs
  with documented <a href="/provenance">provenance</a>.
</Note>

## Turbo (bulk longitude scans)

For century-scale work that evaluates a longitude tens of thousands of times (a
long transit scan, a returns sweep), the `Turbo` tier trades generality for
speed. A turbo pack is a segmented Chebyshev representation of the engine's own
apparent longitude, fit to the engine itself, so one evaluation is a couple of
dozen multiply-adds. The pack is data you mint for your range and bodies; the
runtime evaluator does no I/O and needs no engine.

<CodeBlock
  lang="typescript"
  label="turbo.ts"
  code={`import { Turbo } from "caelus";

// pack: a TurboPack you minted for your range/bodies (python/fit_turbo.py).
const turbo = new Turbo(pack);
turbo.has("mars");            // is this body in the pack?
turbo.longitude("mars", jd);  // apparent longitude (degrees), within [jd0, jd1]`}
/>

<Note>
  Turbo returns longitude only: no latitude, distance, houses, or aspects. Use
  it to find candidate instants fast, then read the full position from the
  engine at those instants. It throws outside the pack's fitted range.
</Note>

## Bundle sizes

The choice between client-side and server-side computation comes down to one
number: the embedded dataset. The engine code tree-shakes small (a chart-only
import drops event search, eclipses, the query and turbo tiers, and the derived
and electional layers); the weight a browser pays is the coefficient data it
bundles.

| What you ship | Approx. gzipped | Notes |
| --- | :---: | --- |
| Embedded dataset (`caelus/data-embedded`) | ~97 KB | planets, Moon series, Pluto, Chiron, nutation, fixed-star catalog; full client-side charts |
| Precise Moon tier (`caelus/node`) | ~700 KB | ~3.1 MB on disk; **lazy**, never in the initial bundle |
| `caelus-wheel` | a few KB | React is a peer dependency, not bundled |
| `caelus-birth` | small code, plus IANA timezone data | the timezone data is the weight, not the resolver |
| `caelus-mcp` | server-side | not a browser bundle |

So: ship `caelus` + `caelus/data-embedded` for full charts in the browser (the
data dominates, the code is a small addition on top of your app), or compute
server-side through the [REST endpoint](/api/chart?lat=27.94&lon=-82.46) or the
[MCP server](/docs/mcp) and ship only `caelus-wheel` to render the result.

<Note>
  Exact per-import figures (minified, gzipped, and brotli) come from
  <code>npm run bundle-sizes</code>, which bundles each entrypoint with esbuild
  the way an app bundler would. Regenerate it after anything that moves the
  package surface; the numbers above are the data tiers, which dominate and do
  not tree-shake.
</Note>
